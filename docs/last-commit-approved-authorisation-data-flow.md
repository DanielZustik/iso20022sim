# Approved Authorisation Last Commit Data Flow

This document explains the code introduced by commit `f592c66` in logical data-flow order rather than file order.

The commit adds an XML authorisation approval path:

```text
raw request XML
-> secure DOM parse
-> request XSD validation
-> selected request fields
-> new approved response DOM
-> response XML string
-> response XSD validation
-> HTTP 200 XML body
```

## 1. Application Bootstrapping

`pom.xml` turns the project into a Spring Boot 3.5.3 Java 17 web app.

`spring-boot-starter-web` provides the HTTP server, controller support, XML-compatible request/response handling, and web application context.

`spring-boot-starter-test` provides the test framework used by the API test, including Spring Boot test support, MockMvc, JUnit, and AssertJ.

`Iso20022SimApplication` is the runtime entry point.

`@SpringBootApplication` tells Spring to scan the `cz.monetplus.iso20022sim` package tree, which discovers the authorisation controller and service.

`main(String[] args)` calls `SpringApplication.run(...)`, creating the Spring application context and starting the web application.

## 2. Service Startup

`AuthorisationService` is marked with `@Service`, so Spring creates it as an injectable application bean.

The service defines the ISO 20022 namespaces used by the data flow:

```java
REQUEST_NS = "urn:iso:std:iso:20022:tech:xsd:caaa.001.001.15"
RESPONSE_NS = "urn:iso:std:iso:20022:tech:xsd:caaa.002.001.15"
```

`REQUEST_NS` is used when reading elements from incoming `caaa.001.001.15` request XML.

`RESPONSE_NS` is used when creating outgoing `caaa.002.001.15` response XML.

`SecureRandom RANDOM` is created once and reused for approval-code generation.

The service stores two schema objects:

```java
private final Schema requestSchema;
private final Schema responseSchema;
```

The constructor loads them from classpath resources:

```java
this.requestSchema = loadSchema("xsd/caaa.001.001.15.xsd");
this.responseSchema = loadSchema("xsd/caaa.002.001.15.xsd");
```

The request schema validates inbound acceptor authorisation requests.

The response schema validates the XML generated by the simulator before it is returned.

`loadSchema(...)` creates a W3C XML Schema factory.

It opens the XSD resource with the class loader.

If the resource is missing, it throws `IllegalStateException`, which fails startup instead of letting the service run without validation.

If the resource exists, it wraps the stream in `StreamSource` and builds a reusable `Schema`.

If the schema cannot be read or parsed, it throws `IllegalStateException` with the original cause.

The request XSD declares a `Document` root that contains `AccptrAuthstnReq`.

The response XSD declares a `Document` root that contains `AccptrAuthstnRspn`.

## 3. HTTP Request Entry

`AuthorisationController` is marked with `@RestController`, so Spring treats returned values as HTTP response bodies.

`@RequestMapping(path = "/api/authorisations")` places all controller endpoints under `/api/authorisations`.

The controller stores an `AuthorisationService`.

The constructor receives the service from Spring dependency injection.

The main endpoint is:

```java
@PostMapping(consumes = MediaType.APPLICATION_XML_VALUE, produces = MediaType.APPLICATION_XML_VALUE)
public ResponseEntity<String> authorise(@RequestBody String requestXml)
```

The endpoint accepts only XML request bodies.

It declares that successful responses are XML.

Spring passes the entire request body as the raw string `requestXml`.

The controller does not inspect or transform the XML itself.

It delegates to:

```java
authorisationService.approveAuthorisation(requestXml)
```

The returned XML string is wrapped in `ResponseEntity.ok(...)`, producing `200 OK`.

## 4. Request Parsing

`approveAuthorisation(...)` starts by parsing the raw XML:

```java
Document requestDocument = parseXml(requestXml);
```

`parseXml(...)` creates a `DocumentBuilderFactory`.

`setNamespaceAware(true)` preserves XML namespaces, which is required because later lookups use namespace plus local element name.

The parser disables DOCTYPE declarations:

```java
documentBuilderFactory.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true);
```

It disables external general entities:

```java
documentBuilderFactory.setFeature("http://xml.org/sax/features/external-general-entities", false);
```

It disables external parameter entities:

```java
documentBuilderFactory.setFeature("http://xml.org/sax/features/external-parameter-entities", false);
```

It blocks external DTD access:

```java
documentBuilderFactory.setAttribute(XMLConstants.ACCESS_EXTERNAL_DTD, "");
```

It blocks external schema access:

```java
documentBuilderFactory.setAttribute(XMLConstants.ACCESS_EXTERNAL_SCHEMA, "");
```

Together, those settings make parsing local and defensive against external XML resource resolution.

The method creates a `DocumentBuilder`.

It parses the request string via `InputSource` and `StringReader`.

If parser configuration, XML syntax, or IO fails, the service throws:

```java
new MalformedXmlException("Request body is not well-formed XML", e)
```

The controller catches `MalformedXmlException`.

The HTTP response for malformed XML is:

```text
400 Bad Request
Content-Type: text/plain
Request body is not well-formed XML
```

## 5. Request Schema Validation

After successful parsing, the raw XML string is validated against the request schema:

```java
validateXml(requestXml, requestSchema, "Authorisation request is not schema-valid caaa.001.001.15 XML");
```

`validateXml(...)` creates a new validator from the supplied schema.

It wraps the XML string in `StringReader`.

It feeds that reader to the validator through `StreamSource`.

If validation fails with `SAXException`, the service throws `SchemaValidationException` using the supplied user-facing message.

If validation fails through an unexpected IO issue, the service throws `IllegalStateException`.

The controller catches `SchemaValidationException`.

The HTTP response for schema-invalid XML is:

```text
422 Unprocessable Entity
Content-Type: text/plain
Authorisation request is not schema-valid caaa.001.001.15 XML
```

## 6. Request Projection

Once the XML is well-formed and schema-valid, the service extracts only the fields needed to build the response:

```java
RequestProjection requestProjection = RequestProjection.from(requestDocument);
```

`RequestProjection` is a Java record with five values:

```java
protocolVersion
exchangeId
transactionDateTime
transactionReference
totalAmount
```

`RequestProjection.from(...)` pulls those values from the request DOM.

It reads `Hdr/PrtcolVrsn` into `protocolVersion`.

It reads `Hdr/XchgId` into `exchangeId`.

It reads `Tx/TxId/TxDtTm` into `transactionDateTime`.

It reads `Tx/TxId/TxRef` into `transactionReference`.

It reads `Tx/TxDtls/TtlAmt` into `totalAmount`.

Each extraction goes through:

```java
firstElementText(requestDocument, REQUEST_NS, localName, fieldLabel)
```

`firstElementText(...)` searches the DOM by namespace and local element name.

If no matching element exists, it throws `SchemaValidationException` with `Missing required field in request`.

If the element exists but has blank text, it throws `SchemaValidationException` with `Field in request is blank`.

Otherwise, it trims the text and returns it.

The resulting projection is the reduced data packet that flows into response generation:

```text
PrtcolVrsn -> response Hdr/PrtcolVrsn
XchgId     -> response Hdr/XchgId
TxDtTm     -> response Tx/TxId/TxDtTm
TxRef      -> response Tx/TxId/TxRef
TtlAmt     -> response Tx/TxDtls/TtlAmt
```

## 7. Response DOM Construction

The service builds the approval response with:

```java
String responseXml = buildApprovedResponse(requestProjection);
```

`buildApprovedResponse(...)` creates a new namespace-aware `DocumentBuilderFactory`.

It creates a `DocumentBuilder`.

It creates a new empty DOM document.

It creates the root element:

```java
Element documentElement = responseDocument.createElementNS(RESPONSE_NS, "Document");
```

That root is appended to the response document.

The response wrapper is appended under `Document`:

```java
AccptrAuthstnRspn
```

The response header is appended under the wrapper:

```java
Hdr
```

The response message function is set to:

```text
MsgFctn = AUTP
```

`AUTP` identifies the generated message as an authorisation response.

The protocol version is copied from the request projection:

```text
PrtcolVrsn = request PrtcolVrsn
```

The exchange id is copied from the request projection:

```text
XchgId = request XchgId
```

The response creation timestamp is generated at runtime:

```java
OffsetDateTime.now(ZoneOffset.UTC).withNano(0).toString()
```

That gives the current UTC time with nanoseconds stripped.

The initiating party is created:

```text
InitgPty/Id = ACQUIRER_SIM
```

That marks the simulator as the party initiating the response.

The authorisation response body is created under the wrapper:

```text
AuthstnRspn
```

An empty environment element is added:

```text
AuthstnRspn/Envt
```

The transaction element is added:

```text
AuthstnRspn/Tx
```

The response transaction id copies request transaction identity:

```text
Tx/TxId/TxDtTm = request TxDtTm
Tx/TxId/TxRef  = request TxRef
```

The response transaction details copy the request amount:

```text
Tx/TxDtls/TtlAmt = request TtlAmt
```

The transaction response branch is created:

```text
TxRspn
AuthstnRslt
RspnToAuthstn
```

The actual approval decision is hard-coded:

```text
Rspn = APPR
```

The authorisation code is generated:

```java
AuthstnCd = generateApprovalCode()
```

`generateApprovalCode()` draws a random integer from `1_000_000` inclusive to `100_000_000` exclusive.

It converts that integer to text.

The result is a 7- or 8-digit approval code.

## 8. Response Element Helpers

`appendElement(...)` creates an XML element in `RESPONSE_NS`.

It appends that element to the supplied parent.

It returns the new child element so the caller can add nested elements below it.

`appendTextElement(...)` calls `appendElement(...)`.

It then sets the element text content.

These helpers keep response creation consistently namespaced.

## 9. Response Serialization

After the response DOM is complete, `buildApprovedResponse(...)` calls:

```java
return toXml(responseDocument);
```

`toXml(...)` creates a `Transformer`.

It sets output encoding to UTF-8.

It enables indentation.

It creates a `StringWriter`.

It transforms the DOM into the writer through `DOMSource` and `StreamResult`.

It returns the serialized XML string.

If serialization fails, it throws `IllegalStateException`.

## 10. Generated Response Validation

Back in `approveAuthorisation(...)`, the generated response is validated before leaving the service:

```java
validateXml(responseXml, responseSchema, "Generated authorisation response is not schema-valid caaa.002.001.15 XML");
```

This catches mistakes in the simulator response structure.

If generated XML is invalid, the same `SchemaValidationException` path is used.

The controller maps that exception to `422 Unprocessable Entity`.

If validation succeeds, the service returns `responseXml`.

The controller returns that XML to the HTTP client as `200 OK`.

## 11. Exception Classes

`MalformedXmlException` is a runtime exception for XML that cannot be parsed at all.

It carries a message and the original parsing cause.

`SchemaValidationException` is a runtime exception for XML that parses but does not satisfy the expected schema or required field extraction.

It also carries a message and an optional cause.

The controller uses those two exception types to distinguish client-facing error status codes.

## 12. Sample Request Data

`approved-authorisation-request.xml` is a complete sample `caaa.001.001.15` request.

Its root namespace matches `REQUEST_NS`.

It contains `Document/AccptrAuthstnReq`.

Its header includes:

```text
MsgFctn = AUTQ
PrtcolVrsn = 1.0
XchgId = 1
CreDtTm = 2026-05-18T12:00:00Z
InitgPty/Id = CARD_ACCEPTOR_SIM
```

Its request body includes merchant, POI, card, context, and transaction data.

The service does not carry all sample fields into the response.

It carries forward only protocol version, exchange id, transaction datetime, transaction reference, and total amount.

## 13. Test Data Flow

`AuthorisationApiTest` starts the full Spring application context with `@SpringBootTest`.

`@AutoConfigureMockMvc` adds `MockMvc`, allowing the test to call the HTTP endpoint without opening a real network port.

The test defines `RESPONSE_NS` so it can read response fields by namespace.

Spring injects `MockMvc`.

The test stores request and response `Schema` objects.

Before each test, `loadSchemas()` creates a schema factory.

It loads `xsd/caaa.001.001.15.xsd` from the test classpath.

It loads `xsd/caaa.002.001.15.xsd` from the test classpath.

The test method loads `samples/approved-authorisation-request.xml`.

It reads the sample bytes as UTF-8 into `requestXml`.

It validates the sample request against the request schema before sending it.

It performs:

```text
POST /api/authorisations
Content-Type: application/xml
Accept: application/xml
Body: sample request XML
```

It expects HTTP `200 OK`.

It expects an XML-compatible response content type.

It extracts the response body as UTF-8 text.

It validates the response XML against the response schema.

It parses the response XML into a DOM.

It extracts `Rspn` from the response namespace.

It extracts `AuthstnCd` from the response namespace.

It asserts:

```text
Rspn == APPR
AuthstnCd is not blank
AuthstnCd length <= 8
```

The test verifies the main contract: a schema-valid authorisation request produces a schema-valid approved authorisation response with an approval code.
